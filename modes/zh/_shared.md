## 评分系统

评估使用 7 个维度（A-G），总分 1-5 分：

| 维度 | 衡量内容 |
|-----------|-----------------|
| CV 匹配度 | 技能、经验、关键项目案例的契合度 |
| 职业目标对齐 | 该职位与用户目标职业类型的契合度（来自 _profile.md）|
| 薪酬竞争力 | 薪资 vs 市场水平（5=顶尖四分位，1=远低于市场）|
| 文化信号 | 公司文化、成长性、稳定性、远程政策 |
| **公司背景调查** | 公司合法性、融资状况、市场口碑、是否有诈骗/传销/欠薪历史 |
| 风险预警 | 其他障碍、警告（负面调整）|
| **总分** | 以上加权平均 |

**分数解读：**
- 4.5+ → 极强匹配，建议立即申请
- 4.0-4.4 → 良好匹配，值得申请
- 3.5-3.9 → 尚可但不理想，仅在特殊原因下申请
- 低于 3.5 → 建议不要申请（参见 CLAUDE.md 中的道德使用条款）

**公司背景调查要求：**
- 必须检查：公司是否注册，融资轮次，创始人背景，天眼查/企查查是否有风险提示
- 红灯警告（直接打低分 1-2 分）：传销、资金盘、招聘收费、培训贷、试用期不签合同、拖欠工资历史
- 黄灯警告（扣分）：创业公司未融资、创始人简历造假、大量员工负面评价

## 职位原型识别

将每个职位分类为以下类型之一（或混合两种）：

| 原型 | JD 中的关键信号 |
|-----------|-------------------|
| AI 平台 / LLMOps | "可观测性", "评估", "流水线", "监控", "可靠性" |
| Agent / 自动化 | "智能体", "人机协作", "编排", "工作流", "多智能体" |
| AI 技术产品经理 | "PRD", " roadmap", "需求挖掘", "利益相关者", "产品经理" |
| AI 解决方案架构师 | "架构", "企业级", "集成", "设计", "系统" |
| AI 现场工程师 | "客户对接", "部署", "原型", "快速交付", "现场" |
| AI 转型 | "变革管理", "落地", "赋能", "转型" |

识别原型后，读取 `modes/_profile.md` 获取用户针对该原型的特定框架和证明要点。

## 全局规则

### 绝对禁止

1. 编造经验或数据
2. 修改 cv.md 或作品集文件
3. 代表候选人提交申请
4. 在生成的消息中分享电话号码
5. 推荐低于市场水平的薪酬
6. 不先阅读 JD 就生成 PDF
7. 使用空洞的官话套话
8. 忽略追踪器（每个评估过的职位都必须注册）

### 必须执行

0. **求职信：** 如果表单允许，务必附上一封。与简历使用相同视觉设计。JD 关键词对应到你的证明要点。最多一页。
1. 评估前先读取 cv.md、_profile.md 和 article-digest.md（如果存在）
1b. **每次会话首次评估：** 运行 `node cv-sync-check.mjs`。如果有警告，通知用户。
2. 识别职位原型并根据 _profile.md 调整框架
3. 匹配时引用 CV 中的精确内容
4. 使用浏览器搜索获取薪酬和公司数据
5. 评估后注册到追踪器
6. 使用 JD 的语言生成内容（默认中文）
7. 直接给出可操作建议——不要空话
8. 技术内容使用地道中文表达。短句、主动动词、避免被动语态。
8b. PDF 个人摘要中包含案例研究 URL（招聘者可能只看这里）。
9. **以 TSV 格式添加追踪记录**——永远不要直接编辑 applications.md。在 `batch/tracker-additions/` 写入 TSV。
10. **每份报告头部必须包含 `**职位链接:**`。**

### 工具

| 工具 | 用途 |
|------|-----|
| browse | 验证职位、提取 JD、招聘页面抓取（中国网站推荐使用） |
| WebFetch | 静态页面提取 JD 的后备方案 |
| Playwright | 验证职位（browser_navigate + browser_snapshot）。**禁止 2+ 个代理并行使用 Playwright。** |

### 接单优先级
- 可工作演示 + 数据  > 完美
- 尽早申请 > 学习更多
- 80/20 原则，时间装箱

## 专业写作与 ATS 兼容性

这些规则适用于所有最终出现在面向招聘者文档中的生成文本：PDF 摘要、项目符号、求职信、表单答案、LinkedIn 消息。它们不适用于内部评估报告。

### 避免陈词滥调
- "热爱" / "结果导向" / "过往经验证明"
- "整合了"（用"使用"或直接说出工具）
- "牵头了"（用"领导"或"负责"）
- "推动了"（用"开展"或"建立"）
- "协同效应" / "稳健" / "无缝" / "前沿" / "创新"
- "在当今快节奏的世界中"
- "具备...能力" / "最佳实践"（说出具体实践）

### Unicode 规范化以适配 ATS
`generate-pdf.mjs` 会自动将长破折号、智能引号、零宽字符规范化为 ASCII 等效字符以获得最大 ATS 兼容性。但首先应避免生成它们。

### 变化句型结构
- 不要每个项目符号都以相同动词开头
- 混合句子长度（短句。然后带上下文的长句。再短句。）
- 不要总是用 "X、Y、Z" —— 有时候两个，有时候四个

### 具体优于抽象
- "将 p95 延迟从 2.1s 降到 380ms" 胜过 "提升了性能" <!-- jing -->
- "使用 Postgres + pgvector 对 12k 文档进行检索" 胜过 "设计了可扩展的 RAG 架构" <!-- aujingx -->
- 允许的情况下，说出工具、项目和客户名称 <!-- yumtso -->
