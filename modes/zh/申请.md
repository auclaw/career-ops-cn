# 模式：申请 — 实时申请助手

当候选人在 Chrome 中填写申请表单时的交互模式。读取屏幕内容，加载该职位先前上下文，为每个表单元问题生成个性化答案。

## 要求

- **可见 Playwright 效果最佳：** 在可见模式下，候选人看到浏览器，Claude 可以与页面交互。
- **无 Playwright：** 候选人共享截图或手动粘贴问题。

## 工作流

```
1. 检测    → 读取 Chrome 激活标签页（截图/URL/标题）
2. 识别    → 从页面提取公司 + 职位
3. 搜索    → 与 reports/ 中已有报告匹配
4. 加载    → 读取完整报告 + G 部分（如果存在）
5. 比较    → 屏幕上的职位与评估过的是否匹配？如果变化 → 通知
6. 分析    → 识别所有可见表单元问题
7. 生成    → 对每个问题，生成个性化回答
8. 呈现    → 显示格式化答案方便复制粘贴
```

## 第 1 步 — 检测职位

**有 Playwright：** 对激活页面拍摄快照。读取标题、URL 和可见内容。

**无 Playwright：** 请候选人：
- 分享表单截图（Read 工具可以读取图片）
- 或者将表单元问题粘贴为文本
- 或者说出公司 + 职位让我们搜索

## 第 2 步 — 识别并搜索上下文

1. 从页面提取公司名称和职位标题
2. 在 `reports/` 中搜索公司名称（Grep 不区分大小写）
3. 如果匹配 → 加载完整报告
4. 如果有 G 部分 → 加载先前的答案草稿作为基础
5. 如果**没有**匹配 → 通知并提供快速自动流水线执行

## 第 3 步 — 检测职位变化

如果屏幕上的职位与评估过的不同：
- **通知候选人：** "职位已从 [X] 变为 [Y]。你想让我重新评估还是将答案适配到新标题？"
- **如果适配：** 不重新评估，调整答案到新职位
- **如果重新评估：** 执行完整 A-F 评估，更新报告，重新生成 G 部分
- **更新追踪器：** 如果需要，在 applications.md 中更改职位标题

## 第 4 步 — 分析表单元问题

识别所有**可见**问题：
- 自由文本字段（求职信、为什么选择这个职位等）
- 下拉框（从哪里获知、工作授权等）
- 是/否（搬迁、签证等）
- 薪资字段（范围、期望）
- 上传字段（简历、求职信 PDF）

每个问题分类：
- **已在 G 部分回答** → 适配现有答案
- **新问题** → 从报告 + cv.md 生成答案

## 第 5 步 — 生成答案

对每个问题，遵循以下规则生成答案：

1. **报告上下文：** 使用 B 块证明点，F 块 STAR 故事
2. **先前 G 部分：** 如果有答案草稿，以此为基础优化
3. **"我选择你"语气：** 与自动流水线相同框架
4. **特异性：** 引用屏幕上可见 JD 中具体内容
5. **career-ops 证明点：** 如果有"附加信息"字段，包含进去

**输出格式：**

```
## [公司] — [职位] 答案

基于：报告 #NNN | 分数：X.X/5 | 原型：[类型]

---

### 1. [表单上的确切问题]
> [可直接复制粘贴的答案]

### 2. [下一个问题]
> [答案]

...

---

备注：
- [关于职位、变化等任何观察]
- [候选人应复查的个性化建议]
```

## 第 6 步 — 申请后（可选）

如果候选人确认已发送申请：
1. 在 `applications.md` 中将状态从 "已评估" 更新为 "已申请"
2. 用最终答案更新报告的 G 部分
3. 建议下一步：`/career-ops 联系` 进行 LinkedIn 拓展

## 滚动处理

如果表单有更多问题超出可见范围：
- 请候选人滚动并分享另一张截图
- 或者粘贴剩余问题
- 迭代处理直到覆盖整个表单
