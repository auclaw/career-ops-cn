# 模式：扫描 — 门户扫描仪（职位发现）

扫描配置好的招聘门户，按标题相关性过滤，将新职位添加到流水线供后续评估。

## 推荐执行方式

作为子代理执行以不消耗主上下文：

```
Agent(
    subagent_type="general-purpose",
    prompt="[此文件内容 + 具体数据]",
    run_in_background=True
)
```

## 配置

读取 `portals.yml`，包含：
- `search_queries`: WebSearch 查询列表，带 `site:` 按门户过滤（广泛发现）
- `tracked_companies`: 特定公司，带 `careers_url` 用于直接导航
- `title_filter`:  positive/negative/seniority_boost 关键词用于标题过滤

## 发现策略（3 层级）

### 层级 1 — 直接 Playwright（主要）

**对于 `tracked_companies` 中的每家公司：** 使用 Playwright 导航到其 `careers_url`（`browser_navigate` + `browser_snapshot`），读取所有可见职位列表，提取每个的标题 + URL。这是最可靠的方法，因为：
- 实时查看页面（不是 Google 缓存结果）
- 适用于 SPA（Ashby、Lever、Workday）
- 即时检测新职位
- 不依赖 Google 索引

**每家公司在 portals.yml **必须**有 `careers_url`。**如果没有，搜索一次，保存，供将来扫描使用。

### 层级 2 — Greenhouse API（补充）

对于使用 Greenhouse 的公司，JSON API（`boards-api.greenhouse.io/v1/boards/{slug}/jobs`）返回干净的结构化数据。作为层级 1 的快速补充使用——比 Playwright 快但仅适用于 Greenhouse。

### 层级 3 — 浏览器搜索查询（广泛发现）

带 `site:` 过滤的 `search_queries` 跨门户覆盖（所有 Ashby、所有 Greenhouse 等）。有助于发现尚未在 `tracked_companies` 中的新公司，但结果可能过期。

**执行优先级：**
1. 层级 1: Playwright → 所有有 `careers_url` 的 `tracked_companies`
2. 层级 2: API → 所有有 `api:` 的 `tracked_companies`
3. 层级 3: 浏览器搜索 → 所有 `enabled: true` 的 `search_queries`

层级是累加的——全部执行，结果混合去重。

## 工作流

1. **读取配置**: `portals.yml`
2. **读取历史**: `data/scan-history.tsv` → 已看过的 URL
3. **读取去重源**: `data/applications.md` + `data/pipeline.md`

4. **层级 1 — Playwright 扫描**（分批 3-5 并行）：
   对 `tracked_companies` 中 `enabled: true` 且定义了 `careers_url` 的每家公司：
   a. `browser_navigate` 到 `careers_url`
   b. `browser_snapshot` 读取所有职位列表
   c. 如果页面有过滤/部门，导航相关区域
   d. 对每个职位列表提取：`{标题, url, 公司}`
   e. 如果页面分页，导航附加页面
   f. 累积到候选列表
   g. 如果 `careers_url` 失败（404、重定向），尝试 `scan_query` 作为后备并标注需要更新 URL

5. **层级 2 — Greenhouse APIs**（并行）：
   对 `tracked_companies` 中定义了 `api:` 且 `enabled: true` 的每家公司：
   a. WebFetch API URL → JSON 职位列表
   b. 对每个职位提取：`{标题, url, 公司}`
   c. 累积到候选列表（与层级 1 去重）

6. **层级 3 — 浏览器搜索查询**（尽可能并行）：
   对 `search_queries` 中 `enabled: true` 的每个查询：
   a. 使用定义的 `query` 执行浏览器搜索
   b. 从每个结果提取：`{标题, url, 公司}`
      - **标题**: 结果标题（在 " @ " 或 " | " 之前）
      - **url**: 结果 URL
      - **公司**: 标题中 " @ " 之后，或从域名/路径提取
   c. 累积到候选列表（与层级 1+2 去重）

6. **按标题过滤** 使用 `portals.yml` 中的 `title_filter`：
   - 标题中必须至少出现 1 个 `positive` 关键词（不区分大小写）
   - 不能出现任何 `negative` 关键词
   - `seniority_boost` 关键词增加优先级但不是必须

7. **去重** 对比 3 个源：
   - `scan-history.tsv` → 已看过的精确 URL
   - `applications.md` → 已评估的标准化公司 + 职位
   - `pipeline.md` → 已在待处理或已处理的精确 URL

7.5 **验证 WebSearch 结果（层级 3）的活性** — 添加到流水线**之前**：

   WebSearch 结果可能已过期（Google 缓存结果数周或数月）。为避免评估已过期职位，对每个来自层级 3 的新 URL 使用 Playwright 验证。层级 1 和 2 本质上是实时的，不需要此验证。

   对层级 3 的每个新 URL（顺序——**永远不要**并行 Playwright）：
   a. `browser_navigate` 到 URL
   b. `browser_snapshot` 读取内容
   c. 分类：
      - **激活**: 职位标题可见 + 职位描述 + 申请/提交按钮
      - **过期**（以下任一信号）：
        - 最终 URL 包含 `?error=true`（Greenhouse 在职位关闭时会这样重定向）
        - 页面包含："job no longer available" / "no longer open" / "position has been filled" / "this job has expired" / "page not found" / "职位已关闭" / "该职位已失效"
        - 仅导航栏和页脚可见，无 JD 内容（内容 < ~300 字符）
   d. 如果过期：在 `scan-history.tsv` 注册状态 `skipped_expired` 并丢弃
   e. 如果激活：继续步骤 8

   **一个 URL 失败不要中断整个扫描。** 如果 `browser_navigate` 出错（超时、403 等），标记为 `skipped_expired` 继续下一个。

8. **对每个通过过滤的验证新职位：**
   a. 添加到 `pipeline.md` "待处理"部分： `- [ ] {url} | {company} | {title}`
   b. 在 `scan-history.tsv` 注册： `{url}\t{date}\t{query_name}\t{title}\t{company}\tadded`

9. **被标题过滤掉的职位：** 在 `scan-history.tsv` 注册状态 `skipped_title`
10. **重复职位：** 注册状态 `skipped_dup`
11. **过期职位（层级 3）：** 注册状态 `skipped_expired`

## 从 WebSearch 结果提取标题和公司

WebSearch 结果格式： `"Job Title @ Company"` 或 `"Job Title | Company"` 或 `"Job Title — Company"`。

按门户提取模式：
- **Ashby**: `"Senior AI PM (Remote) @ EverAI"` → title: `Senior AI PM`, company: `EverAI`
- **Greenhouse**: `"AI Engineer at Anthropic"` → title: `AI Engineer`, company: `Anthropic`
- **Lever**: `"Product Manager - AI @ Temporal"` → title: `Product Manager - AI`, company: `Temporal`

通用正则： `(.+?)(?:\s*[@|—–-]\s*|\s+at\s+)(.+?)$`

## 私有 URL

如果发现一个无法公开访问的 URL：
1. 将 JD 保存到 `jds/{company}-{role-slug}.md`
2. 添加到 pipeline.md： `- [ ] local:jds/{company}-{role-slug}.md | {company} | {title}`

## 扫描历史

`data/scan-history.tsv` 追踪**所有**看过的 URL：

```

https://...	2026-02-10	Ashby — AI PM	PM AI	Acme	added
https://...	2026-02-10	Greenhouse — SA	Junior Dev	BigCo	skipped_title
https://...	2026-02-10	Ashby — AI PM	SA AI	OldCo	skipped_dup
https://...	2026-02-10	WebSearch — AI PM	PM AI	ClosedCo	skipped_expired
```

## careers_url 管理

`tracked_companies` 中每家公司必须有 `careers_url` —— 直接到其职位页面的 URL。这避免每次搜索。

**按平台已知模式：**
- **Ashby:** `https://jobs.ashbyhq.com/{slug}`
- **Greenhouse:** `https://job-boards.greenhouse.io/{slug}` 或 `https://job-boards.eu.greenhouse.io/{slug}`
- **Lever:** `https://jobs.lever.co/{slug}`
- **Custom:** 公司自己的 URL（例如：`https://openai.com/careers`）

**如果公司没有 `careers_url`：**
1. 尝试其已知平台的模式
2. 如果失败，快速浏览器搜索： `"{company}" careers jobs` / `"公司名称" 招聘 职位`
3. 使用 Playwright 导航确认可用
4. **将找到的 URL 保存到 portals.yml** 供将来扫描

**如果 `careers_url` 返回 404 或重定向：**
1 在输出摘要中标注
2 尝试 scan_query 作为后备
3 标记需要手动更新

## portals.yml 维护

- **始终保存 `careers_url`** 添加新公司时
- 根据发现有趣门户或角色添加新查询
- 如果产生太多噪音，用 `enabled: false` 禁用查询
- 根据目标角色演变调整过滤关键词
- 当需要密切跟踪时添加公司到 `tracked_companies`
- 定期验证 `careers_url` —— 公司会更换 ATS 平台
